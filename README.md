# lol-coach

**A personal League of Legends coaching engine — behavioral analysis of my own
ranked games on the climb to Challenger.**

Most stats sites tell you *what* happened (KDA, gold, win rate). lol-coach tries
to tell you *what you should have decided, and why*. It ingests my ranked games,
detects the individual **decision points** inside each one, evaluates the choice
I actually made against the alternatives, and produces a written argument — not a
flag or a number, but context + options with expected value + prose. My own
feedback on those arguments trains a preference model that recalibrates how the
engine weighs each factor.

The bet behind it: climbing out of high elo is a *behavioral* problem
(consistency), not a mechanical one. The engine's job is to surface the mistake
you keep repeating while knowing it's wrong. For my account, the measured
dominant pattern is **over-extending while ahead**.

> Built as a real tool I use on my own account, not a demo. The dashboard and the
> generated coaching prose are in Spanish (I'm the target user); the code and
> docs are in English.

## How it works

```
Riot API + Live Client capture  ->  4 decision detectors  ->  evaluation (HEF)
        ->  options with EV + written argument  ->  human feedback
        ->  Bradley-Terry preference learner  ->  recalibrated weights
```

1. **Capture.** A 1 Hz poller reads the in-game Live Client API for exact
   per-second state (HP, mana, gold, position); Riot's Match-V5 timeline
   backfills the rest. Only ranked games are ingested; normals and remakes are
   tombstoned so they're never re-fetched.
2. **Detect decisions.** Four detectors turn a match into discrete decision
   points: `death` (every death, classified by the real action — over-extension,
   failed disengage, calculated sacrifice, …), `trade` (bad bot-lane damage
   trades in lane), `tempo` (fights the team took without me), and
   `objective_readiness` (was the team actually set up for that dragon/baron?).
3. **Evaluate.** A hierarchical evaluation function (HEF) scores each state on
   `power`, `info`, `action_fit`, and `wave`, and each option gets an expected
   value.
4. **Argue.** Each decision becomes an argument: what I did, what was optimal, and
   why — with the alternative spelled out, never just "you played it wrong". The
   prose is generated by an LLM layer that consumes the structured decision rows.
5. **Learn.** I mark decisions (agree / disagree, and which option was actually
   correct). A Bradley-Terry preference model learns new HEF weights from those
   marks, cross-validated, behind a guardrail — so the engine's judgment tracks
   mine over time.
6. **Measure progress.** A progress view tracks the dominant pattern over a moving
   window with Wilson confidence intervals, so I can answer "am I actually
   improving?" without fooling myself with outcome bias.

## Design decisions worth calling out

These are the parts I'm most proud of:

- **Stdlib-only, on purpose.** Python standard library + `requests`, React over
  UMD as a single `dashboard.html` with no build step, SQLite in WAL mode. No
  FastAPI / Vite / ORM. The constraint keeps the whole system small and
  inspectable; I only add a framework when a concrete pain justifies it.
- **Feedback survives re-analysis.** Re-running the detectors deletes and
  reinserts every decision. Human feedback is preserved by a stable key, and
  marks are validated by a semantic *fingerprint* of the option set rather than a
  brittle index or timestamp — so refactoring a detector doesn't silently throw
  away months of labeling.
- **I/O separated from pure logic** in every detector (`_gather_facts` does all
  the database work; `_evaluate` is pure), which is what makes the criterion
  branches unit-testable without a database.
- **Honest measurement.** Progress is reported with confidence intervals; when the
  intervals overlap, the verdict is "no distinguishable improvement yet" instead
  of a cherry-picked win.
- **176 characterization tests** pin the behavior as a safety net for criterion
  refactors.

## Setup

Requires Python 3.12+ and a free Riot API key.

```bash
# 1. install
python -m venv .venv
.venv\Scripts\python.exe -m pip install -r requirements.txt
.venv\Scripts\python.exe -m pip install -r requirements-dev.txt   # for tests

# 2. configure
copy config.example.toml config.toml
#   then edit config.toml: add your Riot API key (https://developer.riotgames.com)
#   and your account(s). config.toml is gitignored so your key stays local.

# 3. fetch static game data (champion/item data, patch-versioned)
.venv\Scripts\python.exe scripts\download_game_data.py

# 4. run the pipeline, then open the dashboard
.venv\Scripts\python.exe scripts\sync_ascent.py --all
.venv\Scripts\python.exe scripts\dashboard.py        # http://localhost:8765

# tests
.venv\Scripts\python.exe -m pytest
```
